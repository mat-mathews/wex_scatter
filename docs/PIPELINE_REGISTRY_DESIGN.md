## Problem

Scatter needs to know which CI/CD pipelines deploy which projects so it can
answer "if I change this class, which pipelines need to run?" Today this is
handled by a CSV file (`pipeline_to_app_mapping.csv`) parsed with a standalone
script (`parse_pipeline_artifacts.py`). This approach has several problems:

1. **Brittle matching.** Legacy modes look up by solution stem, impact mode
   looks up by project name. Neither is explicitly defined — the CSV just has
   "Application Name" and the consumer hopes it matches.

2. **Flat structure.** One app → many pipelines (nav variants, environments)
   gets flattened into duplicate rows. The 1:N relationship is lost.

3. **Two sources of truth.** `pipeline_to_app_mapping.csv` (generated) and
   `app_to_directory_mapping.json` (manual) overlap but drift.

4. **No staleness detection.** The CSV is generated once and committed. No way
   to know if it reflects the current state of the DevOps repo.

5. **Single source format.** Only parses Azure DevOps YAML with a specific
   `extends.parameters.CIPack.artifactName` structure. No way to add other
   sources without rewriting the parser.

6. **Deployment type discarded.** The artifact prefix (`web_`, `winsvc_`,
   `exe_`) tells you what kind of deployment this is, but the parser throws
   it away.

---

## Design Goals

1. **Canonical format** — one file, one schema, replaces both CSV and JSON
2. **Multi-source ingestion** — pluggable parsers (Azure DevOps YAML today,
   others later) that all produce the same canonical format
3. **Explicit matching keys** — the format declares exactly which project(s)
   a pipeline maps to, eliminating the solution-stem vs project-name ambiguity
4. **Staleness awareness** — the file records when and from where it was
   generated, so users and tooling can tell if it's stale
5. **Graceful unknowns** — when the parser can't resolve a project match, it
   says so explicitly rather than silently dropping the entry
6. **Backward compatible** — scatter continues to work without a pipeline
   registry (empty results, not errors)

## Canonical Format

Two files in `.scatter/` (at the search scope root):

- **`.scatter/pipelines.yaml`** — auto-generated by sync, fully replaceable
- **`.scatter/pipeline-overrides.yaml`** — hand-edited, never touched by sync

Overrides are merged on load, not on write. This means sync can safely
overwrite `pipelines.yaml` without risk of losing manual mappings, and
concurrent syncs from different DevOps repo versions produce trivial merges.
(Priya, Fatima)

### `.scatter/pipelines.yaml`

```yaml
# Auto-generated by: scatter pipelines sync <source-path>
# Manual mappings go in pipeline-overrides.yaml, not here.

version: 1
generated_at: "2026-03-21T14:30:00Z"
content_hash: "a1b2c3d4"          # SHA-256 of applications block for change detection

sources:
  - type: azure-devops-yaml
    label: "cdh-pipelines"        # human-readable, not an absolute path
    parsed_at: "2026-03-21T14:30:00Z"
    files_found: 187
    files_parsed: 133
    files_skipped: 54
    skip_summary:                 # categorized skip reasons (Fatima)
      no_artifact: 31
      yaml_error: 18
      unexpected: 5

applications:
  Lighthouse1.AdHoc.Web.API:
    deploy_type: web
    pipelines:
      - cdh-adhocweb-az-cd
    projects:
      - Lighthouse1.AdHoc.Web.API.csproj
    match_method: exact

  LH1.FileImport.FileLoaderService:
    deploy_type: windows-service
    pipelines:
      - cdh-cdex-fileloader-az-cd
    pipeline_variants:
      pattern: "cdh-cdex-fileloader-nav{N}-az-cd"
      instances:
        - cdh-cdex-fileloader-nav1-az-cd
        - cdh-cdex-fileloader-nav2-az-cd
        # ... all nav instances listed explicitly for lookup
    projects:
      - LH1.FileImport.FileLoaderService.csproj
    match_method: exact

  jobs:
    deploy_type: batch
    pipelines:
      - cdh-batchprocesses-az-cd
    projects: []
    match_method: unresolved
    notes: "Batch pipeline deploys multiple projects. Map individual batch jobs via --app-config-path."
```

### `.scatter/pipeline-overrides.yaml`

```yaml
# Manual pipeline mappings. Merged on load, never overwritten by sync.
# Use this for mappings the parser can't discover automatically.

applications:
  ArchivedDocumentRetrieval:
    deploy_type: service
    pipelines:
      - cdh-summary-archiver-cicd
    projects:
      - ArchivedDocumentRetrieval.csproj
    match_method: manual
```

### Schema Notes

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `version` | int | yes | Schema version for forward compat |
| `generated_at` | ISO 8601 | yes | When the file was last generated |
| `content_hash` | string | yes | SHA-256 of applications block — detects out-of-band edits (Marcus) |
| `sources` | list | yes | Where the data came from (supports multiple) |
| `sources[].type` | string | yes | Parser type identifier (e.g., `azure-devops-yaml`) |
| `sources[].label` | string | yes | Human-readable source name, not an absolute path (Marcus) |
| `sources[].parsed_at` | ISO 8601 | yes | When this source was last parsed |
| `sources[].files_found` | int | yes | Total files found |
| `sources[].files_parsed` | int | yes | Successfully parsed |
| `sources[].files_skipped` | int | yes | Could not parse |
| `sources[].skip_summary` | dict | yes | Categorized skip reasons (Fatima) |
| `applications` | dict | yes | Keyed by application name |
| `applications[].deploy_type` | string | no | `web`, `windows-service`, `batch`, `console`, `service`, `unknown` |
| `applications[].pipelines` | list[str] | yes | Primary pipeline names |
| `applications[].pipeline_variants` | object | no | Variant info (pattern, instance list) |
| `applications[].projects` | list[str] | yes | Matched .csproj filenames (empty if unresolved) |
| `applications[].match_method` | string | yes | `exact`, `case-insensitive`, `manual`, `unresolved` |
| `applications[].notes` | string | no | Human-readable context |

### Deploy Type Mapping

Derived from the artifact name prefix:

| Prefix | deploy_type |
|--------|-------------|
| `web_` | `web` |
| `winsvc_` | `windows-service` |
| `exe_` | `batch` or `console` |
| (none) | `unknown` |

## Architecture: Pipeline Registry Module

Simplified module structure per Sam's feedback — 5 files, not 8:

```
scatter/
  pipelines/
    __init__.py              # public API: load_registry(), sync_registry()
    registry.py              # PipelineRegistry + PipelineApp dataclasses, loader, lookups
    resolver.py              # match application names → .csproj files
    parsers/
      __init__.py            # PipelineParser protocol, RawPipelineEntry, ParseResult, discover
      azure_devops.py        # Azure DevOps YAML parser (replaces parse_pipeline_artifacts.py)
```

### Parser Protocol

Every source format implements this protocol. The protocol, data types, and
parser discovery all live in `parsers/__init__.py`:

```python
from dataclasses import dataclass
from pathlib import Path
from typing import Protocol

@dataclass
class RawPipelineEntry:
    """What a parser extracts from one pipeline definition file."""
    pipeline_name: str           # e.g., "cdh-adhocweb-az-cd"
    artifact_name: str           # e.g., "web_Lighthouse1.AdHoc.Web.API"
    application_name: str        # e.g., "Lighthouse1.AdHoc.Web.API"
    deploy_type: str             # e.g., "web"
    source_file: Path            # which file this came from

@dataclass
class ParseResult:
    """What a parser returns after scanning a directory."""
    entries: list[RawPipelineEntry]
    files_found: int
    files_parsed: int
    files_skipped: int
    skip_reasons: dict[str, int] # categorized: {"no_artifact": 31, "yaml_error": 18, ...}
    errors: list[str]            # human-readable error messages

class PipelineParser(Protocol):
    """Protocol for pipeline source parsers."""

    source_type: str             # e.g., "azure-devops-yaml"

    def can_parse(self, path: Path) -> bool:
        """Return True if this parser can handle the given directory."""
        ...

    def parse(self, path: Path) -> ParseResult:
        """Parse all pipeline definitions in the directory."""
        ...
```

### Adding a New Source Format

To support a new pipeline source (e.g., GitHub Actions, Jenkins, Terraform):

1. Create `scatter/pipelines/parsers/my_format.py`
2. Implement the `PipelineParser` protocol
3. Register it in `scatter/pipelines/parsers/__init__.py`

The sync command auto-discovers parsers. If `can_parse()` returns True, it's used.
If multiple parsers claim the same directory, results are merged.

### Registry Dataclass

```python
@dataclass
class PipelineApp:
    """One application's pipeline mapping."""
    name: str
    deploy_type: str
    pipelines: list[str]
    pipeline_variants: list[str]       # all variant instances, flattened
    projects: list[str]                # matched .csproj filenames
    match_method: str                  # exact, case-insensitive, manual, unresolved

@dataclass
class PipelineRegistry:
    """The loaded, queryable pipeline registry."""
    version: int
    generated_at: str
    content_hash: str
    applications: dict[str, PipelineApp]

    def lookup_by_project(self, project_name: str) -> list[str]:
        """Given a .csproj stem, return pipeline names. Case-insensitive."""
        ...

    def lookup_by_name(self, app_name: str) -> list[str]:
        """Given an application name, return pipeline names. Case-insensitive."""
        ...

    def all_pipeline_names(self) -> list[str]:
        """All known pipeline names, sorted."""
        ...
```

The registry builds reverse indexes on load so all lookups are O(1).

**No `lookup_by_solution()`.** The registry doesn't know about .sln files —
that's scatter's domain model, not the registry's concern. The caller
(v1_bridge) translates solution stems into project names using its existing
solution index, then calls `lookup_by_project()`. (Priya)

### Resolver: Application Name → .csproj

After parsing, the resolver tries to match each `RawPipelineEntry.application_name`
to actual `.csproj` files in the search scope:

1. **Exact match** — `Lighthouse1.AdHoc.Web.API` == `Lighthouse1.AdHoc.Web.API.csproj` stem
2. **Case-insensitive match** — handles `lighthouse1.apps.web.browserdeprecationsite`
3. **Unresolved** — no match found, entry preserved with `match_method: unresolved`

No fuzzy matching in v1. Wait for real data to see if it's needed — YAGNI.
(Anya)

The resolver runs during `sync` when the user has both the DevOps repo and the
.NET codebase available. If the .NET codebase isn't available (e.g., running sync
on a laptop without the monorepo), projects are left empty and resolved later.

## CLI Integration

### New Subcommand: `scatter pipelines`

```bash
# Sync from one or more sources (auto-detects parser)
scatter pipelines sync /path/to/devops-repo/pipelines

# Sync from multiple sources
scatter pipelines sync /path/to/devops-repo /path/to/other-source

# Sync and resolve against the .NET codebase
scatter pipelines sync /path/to/devops-repo --search-scope /path/to/monorepo

# Show current registry status
scatter pipelines status

# List all known pipelines
scatter pipelines list

# Look up pipelines for a specific project
scatter pipelines lookup GalaxyWorks.Data
```

### Sync Output

The sync command prints a diff summary so changes are visible immediately
(Marcus):

```
Pipeline registry sync
  Source: cdh-pipelines (azure-devops-yaml)
  Files: 187 found, 133 parsed, 54 skipped (31 no artifact, 18 YAML error, 5 unexpected)

  Added: 3 applications
  Removed: 1 application (OldService)
  Updated: 5 pipeline assignments
  Unresolved: 12 applications (no matching .csproj)

  Wrote .scatter/pipelines.yaml (133 applications, hash: a1b2c3d4)
```

### Empty Result Safety

If the parser returns 0 applications from a non-empty source directory, sync
refuses to overwrite an existing non-empty registry. Requires `--force` to
proceed. This prevents a broken parser from silently deleting all mappings.
(Fatima)

### Replacing `--pipeline-csv`

The existing `--pipeline-csv` flag continues to work (backward compat) but is
deprecated. The preferred flow:

```bash
# Old way (still works):
scatter --target-project ./MyApp.csproj --search-scope . --pipeline-csv mappings.csv

# New way:
scatter pipelines sync /path/to/devops-repo    # one-time or periodic
scatter --target-project ./MyApp.csproj --search-scope .
# Pipeline data loaded automatically from .scatter/pipelines.yaml
```

Auto-loading: if `.scatter/pipelines.yaml` exists at the search scope root,
scatter loads it automatically — same pattern as the graph cache. (Priya)

### Runtime Match Logging

When scatter loads the registry and runs analysis, log the match rate so users
can tell if the registry is stale or misconfigured (Fatima):

```
Pipeline registry: loaded 133 applications from .scatter/pipelines.yaml
  4 consumers found, 3 matched to pipelines, 1 unmatched (MyUnknownApp)
```

## Consumption Changes

### What Changes in Scatter's Core

The `pipeline_map: Dict[str, str]` that flows through the codebase today gets
replaced with `PipelineRegistry`:

| Current | New |
|---------|-----|
| `Dict[str, str]` (app name → single pipeline) | `PipelineRegistry` (multi-key lookup, 1:N pipelines) |
| `pipeline_map.get(name, "")` → `str` | `registry.lookup_by_project(name)` → `list[str]` |
| Solution stem lookup in v1_bridge | v1_bridge translates stem → project via solution index, then `registry.lookup_by_project()` |
| Consumer name lookup in impact_analyzer | `registry.lookup_by_project(name)` → `list[str]` |

### Data Model Changes

```python
# ConsumerResult: Optional[str] → List[str]
pipeline_name: Optional[str] = None        # old
pipeline_names: List[str] = field(...)     # new

# EnrichedConsumer: str → List[str]
pipeline_name: str = ""                    # old
pipeline_names: List[str] = field(...)     # new
```

### Reporter Changes

- **Console**: `[Pipeline: cdh-example-az-cd]` → `[Pipelines: cdh-example-az-cd, cdh-example-cd]`
- **CSV**: `PipelineName` column → semicolon-separated if multiple
- **JSON**: `"pipeline_name": "..."` → `"pipeline_names": [...]`
- **Pipeline-only output**: unchanged (extracts unique names from list fields)

## Test Strategy (Anya)

### Parser Tests
Use real YAML fixtures in the test directory — not mocks. Cover:
- Standard `extends.parameters.CIPack.artifactName` structure
- Trigger-tag fallback (`resources.pipelines[0].trigger.tags[0]`)
- Tab characters in YAML (replaced before parsing)
- Missing fields, empty files, YAML syntax errors
- Artifact names with and without type prefix (`web_Foo` vs `Foo`)

### Registry Tests
Build a registry in memory, assert:
- `lookup_by_project()` and `lookup_by_name()` return correct pipelines
- Case-insensitive matching works
- Missing keys return empty lists, not None or exceptions
- Overrides merge correctly and take precedence
- Reverse index is consistent

### Resolver Tests
Given a list of `RawPipelineEntry` names and a list of `.csproj` stems, assert:
- Exact matches categorized correctly
- Case-insensitive matches categorized correctly
- Unresolved entries preserved with `match_method: unresolved`

### Round-Trip Test
Sync → write YAML → load YAML → assert registry contents match original
parse results. Catches serialization bugs early.

### Backward Compatibility Test
`--pipeline-csv` still works. Load a CSV, run analysis, assert pipeline names
appear in results identically to today. This is the regression gate for Phase 2.

## Migration Path

1. **Phase 1: Module structure + canonical format + Azure DevOps parser**
   - Create `scatter/pipelines/` module (5 files)
   - Port `parse_pipeline_artifacts.py` logic into `parsers/azure_devops.py`
   - Implement `PipelineRegistry` with multi-key lookup
   - `scatter pipelines sync` command writes `.scatter/pipelines.yaml`
   - Auto-load registry if file exists
   - Tests: parser fixtures, registry lookups, round-trip

2. **Phase 2: Wire into scatter core**
   - Replace `Dict[str, str]` with `PipelineRegistry` in ModeContext
   - Update v1_bridge: translate solution stem → project name via solution
     index, then call `registry.lookup_by_project()` (Priya)
   - Update impact_analyzer to use `registry.lookup_by_project()`
   - Update data models (`pipeline_name` → `pipeline_names`)
   - Update all reporters
   - Deprecate `--pipeline-csv` (still works, loads into registry format)
   - Tests: backward compat, reporter output changes

3. **Phase 3: Resolver + staleness**
   - Project resolution during sync (`--search-scope`)
   - Staleness warning (configurable threshold, default 7 days)
   - `scatter pipelines status` command
   - Runtime match logging (Fatima)
   - Content hash comparison for change detection (Marcus)

4. **Phase 4: Additional parsers** (as needed)
   - GitHub Actions YAML
   - Jenkins
   - Other formats

## Design Decisions (from team review)

| Decision | Rationale | Who |
|----------|-----------|-----|
| Overrides in separate file (`pipeline-overrides.yaml`) | Sync can safely overwrite generated file without losing manual mappings. Concurrent syncs produce trivial merges. | Priya, Fatima |
| No `lookup_by_solution()` on registry | Registry shouldn't know about .sln files. Caller translates solution stem → project name using its own solution index. Keeps registry clean. | Priya |
| `.scatter/` at search scope root | One location, predictable. Same as graph cache. | Priya |
| `content_hash` in YAML | Detect out-of-band edits. Compare on load to tell "unchanged" vs "modified outside sync." | Marcus |
| `sources[].label` not absolute path | Absolute paths are machine-specific. Label is human-readable and portable. | Marcus |
| Sync prints diff summary | "Added 3, removed 1, updated 5" — visible changes, not just a count. | Marcus |
| Refuse empty-result overwrite | Parser returning 0 from non-empty source = broken parser, not empty registry. Require `--force`. | Fatima |
| Categorized skip reasons in YAML | "54 skipped" isn't actionable. "31 no artifact, 18 YAML error" is. | Fatima |
| Runtime match logging | "4 consumers, 3 matched, 1 unmatched" tells you if the registry is stale. | Fatima |
| No fuzzy matching in v1 | Wait for real data. Exact + case-insensitive covers known cases. Fuzzy risks false positives. | Anya |
| Real YAML fixtures in tests | Don't mock the YAML parser. Parse real YAML to catch real edge cases. | Anya |
| Round-trip test | sync → write → load → assert. Catches serialization bugs early. | Anya |
| 5 files not 8 | Merge loader into registry, sync into `__init__`, protocol into `parsers/__init__`. Split later if needed. | Sam |

## What We Don't Know Yet

These will be answered when the DevOps config repo is accessible:

1. **Is `extends.parameters.CIPack.artifactName` consistent?** The existing
   parser has a fallback (`resources.pipelines[0].trigger.tags[0]`), suggesting
   not all files follow the same structure. How many variants are there?

2. **Is the artifact name the only link to .csproj?** Or do some YAML files
   contain project paths, solution references, or directory mappings?

3. **How are nav/environment variants structured?** Is `nav1`, `nav2` etc.
   encoded in the filename, in YAML parameters, or both?

4. **Are there pipelines that don't map to any project?** Infrastructure
   pipelines, shared utility pipelines, etc.

5. **How often does the mapping change?** Daily? Weekly? Only on big reorgs?
   This determines how aggressive staleness warnings should be.

The parser protocol and resolver are designed to absorb these answers without
architectural changes. The Azure DevOps parser can grow new extraction paths.
The resolver can learn new matching heuristics. The canonical format handles
unknowns via `match_method: unresolved`.

## Risk

**Low for Phase 1.** The new module is additive — it doesn't change any existing
behavior until Phase 2 wires it in. The parser protocol means we can iterate on
the Azure DevOps parser as we learn more about the YAML structure without
changing the rest of the system.

**Medium for Phase 2.** Changing `pipeline_name: str` to `pipeline_names: List[str]`
touches every reporter and both analysis paths. Mitigated by: tests exist for
all reporters, the v1_bridge, and the impact analyzer. Backward compat test
for `--pipeline-csv` provides the regression gate.
